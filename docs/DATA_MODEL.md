# Bridge OS — Data Model

> Updated in **Phase 1**. Each subsequent phase appends.

## Entity overview (Phase 1)

| Entity | Purpose | Key fields |
|---|---|---|
| **Patient** | Thalassemia patient on a Blood Bridge | `name`, `age`, `blood_group`, `kell_negative`, `city`, `hospital`, `transfusion_cadence_days`, `last_transfusion_date` |
| **Donor** | Voluntary blood donor in the network | `name`, `age`, `blood_group`, `kell_negative`, `phone`, `preferred_language`, `city`, `last_donation_date`, `response_rate` |
| **Bridge** | The cohort that ties one patient to ~10 donors | `patient_id` (unique), `name`, `status`, `created_at` |
| **BridgeMembership** | Donor's membership in a bridge | `bridge_id`, `donor_id`, `role`, `status`, `joined_at` |

### Relationships

- `Patient` 1 ⟷ 1 `Bridge` (one patient, one active bridge)
- `Bridge` 1 ⟷ N `BridgeMembership` (one bridge has ~8–10 donors)
- `Donor` 1 ⟷ N `BridgeMembership` (a donor can be in multiple bridges over time, though typically one)
- Unique constraint on `(bridge_id, donor_id)` to prevent duplicate memberships

### Enums

- `BloodGroup`: `O+`, `O-`, `A+`, `A-`, `B+`, `B-`, `AB+`, `AB-`
- `Language`: `en`, `hi`, `te`, `ta`, `mr`, `bn`, `kn`, `gu`
- `BridgeStatus`: `active`, `paused`, `archived`
- `MembershipStatus`: `active`, `paused`, `exited`
- `MembershipRole`: `primary`, `backup`
- `BridgeHealth`: `stable` (≥8 active donors), `at_risk` (5–7), `critical` (<5)

> The `BridgeHealth` enum is the *target* of the Phase 4 stability model. In
> Phase 1 it's stubbed from the active donor count. Phase 4 replaces the stub
> with XGBoost-predicted churn risk over the cohort.

### Derived fields

| Property | Defined on | Logic |
|---|---|---|
| `Patient.next_transfusion_date` | Patient | `last_transfusion_date + transfusion_cadence_days` |
| `Patient.days_until_transfusion` | Patient | `(next_transfusion_date - today).days` |
| `Donor.days_since_donation` | Donor | `(today - last_donation_date).days` |
| `Donor.is_eligible_to_donate` | Donor | active and ≥90 days past last donation |
| `Bridge.active_donor_count` | Bridge | count of memberships with `status=active` |
| `Bridge.total_donor_count` | Bridge | total memberships |
| `Bridge.health` | Bridge | from active_donor_count (Phase 1) → ML model (Phase 4) |

### Cross-dialect types

`app.models.types.GUID` is a `TypeDecorator` that uses Postgres `UUID` in
production and `CHAR(36)` in SQLite tests. All primary keys use this so the
same models drive both runtimes.

## Synthetic data calibration

The generator at `app.synthetic.generator` produces a default dataset of:

- 50 patients (60% Hyderabad-weighted, all Rh+ for Phase 1)
- 500 donors (Indian-realistic blood-group distribution)
- 50 bridges (one per patient)
- 8–10 active donors per bridge, ABO/Rh-compatible with the patient
- Special featured patient: **Aarav Reddy** (8 yo, B+, Kell-negative,
  Apollo Hyderabad) with **Priya Sharma** as a destabilizing cohort member
  (42% response rate, 38h avg response) — used in the demo flow.

To reseed:

```powershell
cd backend
.venv\Scripts\activate
python -m scripts.seed
```

## Phase 2 additions

No new entities — Phase 2 surfaces existing `Donor` data via dedicated endpoints
and adds a derived field `bridge_count` (count of active memberships per
donor) that the donors-list endpoint computes in a single `GROUP BY` query.

## Phase 3 additions

No new entities. Phase 3 adds two server-computed projections on `Patient`:

- `has_bridge`, `bridge_health`, `active_donor_count` — surfaced on
  `PatientListItem` by joining with the patient's bridge and using
  `Bridge.health` (Phase 1 stub).
- `projected_transfusions` — array of the next 6 calendar dates computed
  from `last_transfusion_date + n × transfusion_cadence_days`. Pure derivation,
  no DB writes. Replaced by the OR-Tools solver output in Phase 5.

## Phase 4 additions (ML)

No new persisted entities — the stability predictor reads existing rows.
Adds three on-disk artifacts under `backend/data/models/`:

- `stability_30d.json`, `stability_60d.json`, `stability_90d.json` — XGBoost models in their native JSON format
- `stability_report.json` — training metadata (train/test AUC, Brier, positive rate, feature names, trained-at timestamp)

These are generated by `python -m scripts.train_stability` and loaded
on first endpoint call (singleton via `lru_cache`).

## Future entities (planned per phase)

- **Phase 6** — `Recommendation` ranked candidates
- **Phase 10** — `WhatsAppMessage` log

## Phase 5 additions (scheduler)

No new persisted entities. The rotation is computed live by the CP-SAT
solver on every `/schedule` request — solve time is ~50 ms for the
365-day horizon × 8 donors, so caching isn't worth it yet. (Phase 7
analytics will cache aggregate statistics across all bridges, but
individual rotations stay live-computed for the demo's "re-solve" loop.)

## Phase 10 additions (WhatsApp)

| Entity | Purpose | Key fields |
|---|---|---|
| **WhatsAppMessage** | One message in a donor conversation | `direction` (`inbound`/`outbound`), `from_number`, `to_number`, `body`, `status`, `twilio_sid`, `template_key`, `donor_id`, `bridge_id`, `created_at` |

### Relationships

- `WhatsAppMessage` N ⟷ 1 `Donor` (nullable; inbound from unknown numbers leaves it `null`) — `ON DELETE SET NULL`
- `WhatsAppMessage` N ⟷ 1 `Bridge` (nullable; only set when message is contextual to a bridge, e.g. template send) — `ON DELETE SET NULL`

### Enums

- `MessageDirection`: `inbound`, `outbound`
- `MessageStatus`: `queued`, `sent`, `delivered`, `read`, `received`, `failed`, `mocked`

The `mocked` status is reserved for messages generated when Twilio is not
configured — the same row shape is used so the UI doesn't branch.

## Phase 11 additions (Care Agent)

| Entity | Purpose | Key fields |
|---|---|---|
| **AgentMessage** | One message in a Care Agent conversation | `session_id`, `role` (`user`/`assistant`/`system`), `content`, optional `donor_id`/`bridge_id`/`patient_id`, `language`, `provider`, `model`, `tokens_in`, `tokens_out`, `created_at` |

### Notes

- Sessions are **not** a separate table — `session_id` is just a UUID that ties
  messages together. First turn generates it; subsequent turns pass it back.
- The optional entity FK fields are denormalised (no `ON DELETE` cascade) —
  conversations should survive deletion of the underlying donor/bridge for
  audit purposes.
- Provider columns (`provider`, `model`, `tokens_in`, `tokens_out`) are filled
  on assistant rows only; user rows leave them null.
- "Cohort memory" is **not stored** as embeddings in this version — it's
  reassembled fresh on each turn from `build_donor_context()` /
  `build_bridge_context()` / `build_patient_context()`. Adding a pgvector
  table later is straightforward: serialise each context blob, embed once,
  query top-K on each turn.

## Phase 14b additions (cohort memory)

| Entity | Purpose | Key fields |
|---|---|---|
| **CohortMemory** | Episodic note + embedding for RAG over past turns / events | `kind` (`donor`/`bridge`/`patient`/`wa_thread`/`agent_qa`/`recruit`), `entity_id` (nullable), `summary` (text), `embedding` (JSON list of floats), `embedding_provider`, `embedding_dim`, `created_at` |

### Notes

- `embedding` is JSON so the same column works on SQLite and Postgres without
  a dialect-specific type. To upgrade to native pgvector in production, change
  the column to `Vector(dim)` and rewrite `app/agent/memory.py::retrieve_memories`
  to use `ORDER BY embedding <-> :q LIMIT k`. The public function signature and
  the API responses stay the same.
- `embedding_provider` records which model produced the vector — `local`,
  `openai`, or `bedrock_titan`. The retriever skips rows with mismatched
  `embedding_dim` so swapping providers degrades gracefully instead of
  exploding on dot-product errors.
- Rows are written automatically: every `POST /agent/chat` records the Q&A
  as an `agent_qa` memory. (WhatsApp sends and recruit confirmations could
  emit `wa_thread` / `recruit` memories the same way — wiring left for a
  future polish pass.)
